Contexts

Uses

By default, a database update runs all undeployed s in the . If you add a context to a and use a context filter in the CLI, the update only runs that match the context filter. If you don't specify a context filter in the CLI, every undeployed in your runs, even if they have context attached.

For example, if you want to tag your s per environment and run only these s, set a context on each and specify a matching value for --context-filter in the CLI.

Syntax

If you use versions 4.16.0 to 4.23.0, use the syntax contextFilter instead of context in your s. However, context is still a supported alias.

Context logic

In 4.23.0+, you can specify a context using @ in addition to AND, OR, !, and parentheses in the s. This requires you to designate a context or label explicitly when is run for the to run. The attribute context: @test causes a to not run if runs without any contexts provided.

  • context="@test"

In 3.2.0+, you can specify a context using AND, OR, !, and parentheses in the s. Without parentheses, the order of operations is !, AND, and then OR. For example:

  • context="!test"
  • context="v1.0 or map"
  • context="!qa and !main"

Using a "," to separate contexts works like an OR operation (a comma is an alias for OR):

  • "test, qa" is the same as "test OR qa"
  • "test, qa and main" is the same as "(test) OR (qa and main)"

Running contexts

If you use 4.23.0 or earlier, use the syntax --contexts instead of --context-filter.

You can only specify context filtering logic in a definition, but you can still specify multiple contexts when running in the CLI. However, you can only list out all the contexts that apply to the current run.

liquibase update --context-filter="test" --changelog-file=example-changelog.xml

If your includes several s with complex and simple context filters such as context="qa and main and !dev" for 1 and context="test" for 2, you need to pass the following on the command line to deploy them:

liquibase update --context-filter="test,qa,main" --changelog-file=example-changelog.xml

Using contexts for test data

If you manage your test data with , it is best practice to have this data in line with all your other s, but marked with a "test" context. When you want your test data inserted, run a database update and specify the "test" context in the CLI. When you need to migrate your production database, don't include the "test" context, and your test data will not be included.

If you do not specify any contexts in the CLI at runtime, every will be applied, including those marked with a "test" context.

If you have multiple test environments or test data sets, simply tag them with different contexts, like "min-test" and "integration-test".

Using contexts to control test data is better than having a separate s tree because later s and changes will be applied to existing test data the same as they are applied to production data. If you had a set of test data that was created and simply added after the database is set up, you would be constantly manually updating your test data scripts to keep them in line with the current database schema.

Multi-DBMS s

If you need to use one for multiple databases, and you only want specific s to run on each database, it is possible to use the context tag to filter them, and then run liquibase update --context-filter="<dbname>" in your command line. However, this is not recommended. For example:

<changeSet  id="1-lawful-evil"  author="adrian" context="postgres">
    <createTable  tableName="my_table">
        <column  name="id"  type="int"/>
    </createTable>
</changeSet>

Instead, it is a best practice to use the dbms tag to differentiate s by database type, and then run liquibase update in your command line. This is a clearer use of contexts and decreases the possibility of errors. You can use context and dbms on the same , but only dbms should refer to your database type. For example:

<changeSet  id="1-lawful-good"  author="adrian" dbms="postgres">
    <createTable  tableName="my_postgres_table">
        <column  name="id"  type="int"/>
    </createTable>
</changeSet>
<changeSet  id="2-lawful-good"  author="adrian" dbms="oracle">
    <createTable  tableName="my_oracle_table">
        <column  name="id"  type="int"/>
    </createTable>
</changeSet>
<changeSet  id="3-lawful-good"  author="adrian" dbms="oracle" context="test">
    <createTable  tableName="my_oracle_test_table">
        <column  name="id"  type="int"/>
    </createTable>
</changeSet>

Default context

You can specify a context in the root node to assign that context to all s in the by default.

The specified context will have AND with any context specified in s within the file.

include and includeAll contexts

You can specify the context in <include> or <includeAll> tags. If specified, the given context is added to all s in the included file(s).
