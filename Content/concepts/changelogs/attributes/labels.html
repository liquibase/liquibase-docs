Labels

Uses

You can typically use labels to:

  • Manage and track your deployments.
  • Apply complex logic to filter the s eligible for deployment.
  • Mark s based on features and then selectively apply those features at the execution time instead of using version control branches.
  • Use different languages for marking s. For example, you can mark your s both with Spanish and English.
  • Assign ticket numbers to particular s to track and identify their purpose, such as JIRA ticket numbers

label-filter runtime syntax

In 4.23.0+, you can specify a label using @ in addition to AND, OR, !, and parentheses in the changesets. The @ is used for labels only at run time and forces any specific labeled to execute.

If a is labeled as lables=test, only this will be forced to execute if at runtime we specify --label-filter=@test.

If you have a marked with a label that starts with @ and you run without specifying the --label-filter attribute, the changeset will not be executed. If label filters (--label-filter=) are specified as @test at runtime as described in the example above, no will be executed if there are not any labels titled test.

Label filters can be specified using @, AND, OR, !, and () (parentheses that are used for grouping). For example:

  • --labelFilter="@test"
  • --labelFilter="!v.0.1"
  • --labelFilter="v.1.0 and v.1.1"
  • --labelFilter="v.1.0 or v.1.1"
  • --labelFilter="!v.1.0 and !v.1.1"

Using "," to separate labels works like an OR operation (a comma is an alias for OR). For example:

  • --label-filter="v.1.0, v.1.1" is the same as --label-filter="v.1.0 OR v.1.1"
  • --label-filter="v.0.1, v.1.0 and v.1.1" is the same as --label-filter="(v.0.1) OR (v.1.0 and v.1.1)"

You can run the --label-filter to determine which s in the to evaluate based on its label. Learn more about the labelFilter logic at this blog post.

  • In your , you can specify only a simple list of labels to apply to the . For example, labels="v.0.1, v.1.0".
  • At runtime, you can specify a complex expression with the labels that you want to execute. For example, --labelFilter="!v.0.1" or --labelFilter="v.1.0 or (v.1.1 and v.1.2)".

Prior to version 4.16.0, the CLI argument was --labels rather than --label-filter. In 4.16.0 and beyond, --labels is a supported alias for --label-filter.

Usage

It is best practice to enumerate your s or describe what a is used for. An example of labels indicating the version or a specific feature can be "1.0" or "shopping_cart". In this case, labels will allow you to run changes with:

  • --label-filter=1.0 to deploy the 1.0 s or --label-filter=shopping_cart to deploy the s related to the shopping cart.
  • --label-filter="1.0 or (1.1 and shopping_cart)" to deploy the 1.0 s and only the 1.1 features related to the shopping cart.
  • --label-filter="1.0 or (1.1 and !shopping_cart)" to deploy the 1.0 s and the 1.1 features that are not related to the shopping cart.

You can see an example of running the update command with complex --label-filter statements:

liquibase --output-file=update.txt update --changelog-file=changelog.xml --label-filter="1.0 or (1.1 and !shopping_cart)"

If a doesn't have a label, it will always run. If you make a deployment without specifying any labels in the label filter, all s will run.

If you run the liquibase update --label-filter=1.0 command, it will deploy all s with the label 1.0 and all s without any label. If you run liquibase update, it will deploy all s, whether they have labels or not.

include and includeAll labels

You can specify the labels in include and includeAll tags. If specified, the given label is added to all s in the included file(s).
