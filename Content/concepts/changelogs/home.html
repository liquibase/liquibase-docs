Changelog

With , you use a text-based file to sequentially list all changes made to your database. This ledger helps audit your database and execute any changes that are not yet applied. You can store and version your in any source control tool.

An individual unit of change in your is called a Changeset. When you want to modify your database, simply add a new and specify its operation as a Change Type. For example, you may add a to create a new table, and another to drop a primary key.

You can also include other s in a main , minimizing conflicts between different teams or workflows. You can also specify preconditions, contexts, labels, and other s in your to precisely control which s run—and in which environments.

When you run the update command, deploys the changes you specify in your to your database. You can specify which you want to use with the --changelog-file argument in your command line (CLI), your file, as an environment variable, or using the JAVA_OPTS Environment Variable.

To learn more about using s and s, see Introduction to [%=General.Liquibase%]. If you understand the basics of and want to learn how to structure your s, see Design Your [%=General.Liquibase%] Project.

File formats

Your s can be written in the SQL, XML, YAML, or JSON formats. determines the format by the file extension and the header. If you are using the include or includeAll tags in your root to reference other s, your root must be an XML, YAML, or JSON file. Click the drop-downs to see examples.

The following examples demonstrate the structure of a using generic terms. For examples that use real change types and preconditions, see the "Example Changelogs" pages linked in the drop-down tabs.

Runtime logic

When you run a database update, the migrator detects the file type from the file extension and then parses the header. checks any global preconditions specified. If any of the global preconditions fail, exits with an error message explaining what failed. Preconditions can be typically used for both documenting and enforcing expectations or assumptions. For example, you can specify the DBMS to be run against the or the user you should log in to run changes.

If all preconditions are met, will begin running and include tags in the order they appear in the file. You can control which s run in a particular migration using contexts, labels, and other s on the level. uses the [%=General.databasechangelog%] table to keep track of which changes it has already deployed. In 4.27.0 and later, also records additional migration information in the [%=General.databasechangeloghistory%] table.

Each contains the id and author tags. The id tag, author tag, search path location, and name of the file create a unique identifier for the .

The XML schema for the tag is available at: http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-latest.xsd. The version of the XSD can be set to either latest to match your current version of , or latest can be replaced with a specific version, like http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.25.xsd. Legacy XSDs are listed on the XML Format page.
