Changeset

A is the basic unit of change in . You store all your s in your Changelog. Your s contain [%=General.changetypes%]s that specify what each change does, like creating a new table or adding a column to an existing table.

A is uniquely tagged by both the author and id s (author:id), as well as the file path. The id tag is just an identifier—it doesn't direct the order that changes are run and doesn't have to be an integer. To deploy the with the update command, you must include both author and id. You can specify Preconditions, Contexts, Labels, and other s in individual s to control exactly when they run.

It is a best practice to specify only one type of change per . Doing so avoids failed auto-commit statements that can leave the database in an unexpected state. When you deploy your changes, each either succeeds or fails; if it fails, you can correct it and deploy again. You can also add comments to individual s to explain why they're important.

File formats

The format of your depends on the file type of your , which can be SQL, XML, YAML, or JSON. For examples of entire s in each format, see the pages linked in the following drop-down tabs.

SQL example

Read more: Example Changelogs: SQL Format.

--changeset nvoxland:1
create table company (
    id int primary key,
    address varchar(255)
);
 XML example

Read more: Example Changelogs: XML Format.

<changeSet  id="1"  author="nvoxland">
    <createTable  tableName="company">
        <column  name="address"  type="varchar(255)"/>
    </createTable>
</changeSet>
 YAML example

Read more: Example Changelogs: YAML Format.

databaseChangeLog:
  -  changeSet:  
      id:  1
      author:  nvoxland
      changes:
        -  createTable:
            tableName:  company
            columns:
              -  column:
                  name:  address
 JSON example

Read more: Example Changelogs: JSON Format.

{
  "changeSet": {
    "id": "1",
    "author": "nvoxland",
    "changes": [
      {
        "createTable": {
          "tableName": "company",
          "columns": [
            {
              "column": {
                "name": "address"
              }
            }
          ]
        }
      }
    ]
  }
}

For other formats, see Example Changelogs: Other Formats.

Runtime logic

When you run a database update, reads the s in your in order. For each one, it checks the [%=General.databasechangelog%] table to see if the combination of id/author/filepath has already been deployed.

If a has already been deployed to your database, skips it unless it has a runAlways=true tag. After running all the changes in the s, inserts a new row with the id/author/filepath along with an MD5Sum (checksum) of the in the . For more information, see Changeset Checksums.

filepath is the path that defines the changelog-file . Even if the same file is referenced with a different path, that is considered a different file unless the logicalFilePath is defined.

attempts to execute each in a transaction that is committed at the end, or rolled back if there is an error. Some databases auto-commit statements, which interferes with this transaction setup and could lead to an unexpected database state. Therefore, it is best practice to have just one change per unless there is a group of non-auto-committing changes that you want to apply as a transaction such as inserting data.
