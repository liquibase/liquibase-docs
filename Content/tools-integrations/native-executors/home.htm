Native Executors

primarily uses JDBC drivers to connect and deploy changesets to target databases, but there are instances when the JDBC driver is not sufficient to process highly specialized SQL.

extends the ability to deploy changesets using native database clients so during execution time, a native executor is invoked for changesets that request it to deploy a payload to a target database.

For certain commands and administrative tasks will always use the JDBC driver.

Why Should I Care?

Native database clients have been in use for a long time. Think "sqlplus" for Oracle or "sqlcmd" for SQL Server database. Developers and DBA's have become familiar with these tools and can adapt their SQL code for their database client.

Experienced users understand the output of these tools and expect execution logs detailing the deployed SQL when requesting database deployments.

These details are not visible when deploying with a JDBC driver.

Running SQL with

The update command deploys the SQL payload and queries the database to understand what it needs to deploy based on previous deployments.

can perform SQL payload deployment using either a JDBC driver or a native executor.

With , deciding whether to use a JDBC driver, or native executor is configurable by the developer who authors the database changesets. The author can add a tag on the changeset level to use a native executor. Without this tag, will default to using the JDBC driver.
