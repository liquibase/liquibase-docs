Native Executors

primarily uses JDBC drivers to connect and deploy s to a database. However, there are instances when the JDBC driver is insufficient to process highly specialized SQL.

Using native executors with can provide distinct advantages, such as the ability to process database client-specific settings and environment variables within the SQL script.

always uses the JDBC driver for certain commands and administrative tasks.

Compatible Native Executors

Four native executors can be used with :

  • Oracle: use the sqlplus native executor.
  • Microsoft SQL Server: use the sqlcmd native executor.
  • PostgreSQL: use the psql native executor.
  • MongoDB Pro: use the mongosh native executor.

Database native executors are not shipped with . Users need to install the compatible native executor specific to their database type.

Using Native Executors

The runWith 

decides whether to use the native executor based on the presence of the runWith attribute. You must apply this to each you want to deploy using a native executor:

  • In Formatted SQL s, you can apply runWith to any .
  • In XML, YAML, and JSON s, you can only apply runWith to s using the sql and sqlFile s.

runWith is applied during update and rollback operations. When runWith is not present, uses the JDBC driver to deploy the .

To learn more about using the runWith  with native executors, refer to the database-specific native executor page linked in Compatible Native Executors.
