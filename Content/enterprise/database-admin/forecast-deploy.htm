Implementation Guide: Forecast and Deploy

's "Forecast" is a necessary operation prior to deploying changes. In automation, you would need a separate "Forecast" job which allows you to simulate a deployment. Typically, a release engineer would run the forecast job days before performing a deployment to a target environment. This would create a forecast report - an HTML report - which could be shared with the team to ensure all the correct changes are being forecasted. This also allows the team to make any necessary changes and rerun forecast job in preparation for deployment.

It is preferred that in automation, you always perform a "Forecast" prior to performing a "Deploy" operation. This ensures best practice from automation perspective as you will always have a Forecast report available for every Deploy operation.

By default, Datical will perform a "forecast" prior to deploy. This is controlled by project settings called Deployment Mode. You can change this mode to "Quick Deploy" in which case "Forecast" operation is skipped.

Inputs required

You may decide to create one Deploy job and allow user to select the target database environment to deploy. Or you may decide to create separate Deploy jobs for each database environment. For example, "Deploy to QA", "Deploy to STAGE" and "Deploy to PROD".

Instructions here will be for creating a single Deploy job controlled by user selection.

Environment Variables

In this automation job, you will be issuing the "hammer" command several times. It is best to add the path to "hammer" command in your PATH environment variable. 

  • "hammer" is located in <datical_install_dir>/repl directory.

Step-by-step guide

Follow these steps to setup a Deploy job in your automation tool:

  1.  Setup a user selection parameter for each database environment. We will call this "DBDEF" parameter. 
  2. Start with a clean workspace. Your automation tool may already have a template step for performing a clean operation.
  3. Fetch the desired version of the Datical artifact which contains your database changes. This is the same artifact version which you will later use to deploy to the target database environment.
    1. If you don't use artifacts, you may also checkout the Datical project from your SCM such as Git or TFS. This is not a typical DevOps workflow, but could be necessary in certain other workflows.
  4. If your artifact is a zip file, then unzip your artifact. The artifact is a complete project which contains all your database changes that have been packaged.
    1. For running a Deploy job you will need to be in the project directory.
  5. "cd" into the project directory
  6. First, issue the Forecast command using the user supplied "DBDEF" parameter. Here is what your Forecast command may look like - this may change based on your operating system of your automation agent:
    1. hammer forecast %DBDEF%
    2. Optionally, you can also pass "--genSQL" argument if you would like Datical to automatically generate the SQL script for all the changes that are forecasted to be deployed. 
  7. It is desirable to perform a status prior to deployment. This will allow you to capture what is already deployed to the target database:
    1. hammer statusDetails %DBDEF%
  8. Issue the Deploy command using the user supplied "DBDEF" parameter. Here is what your Deploy command may look like - this may change based on your operating system of your automation agent:
    1. hammer deploy %DBDEF%
    2. Optionally, you can also pass "--genSQL" argument if you would like Datical to automatically generate the SQL script for all the deployed changes. 
  9. It is desirable to perform a status after the deployment. Together with status prior to deployment, you will have the before and after status of the database.
    1. hammer statusDetails %DBDEF%

Here is what your series of "hammer" commands may look like:

set PATH=%PATH%;C:\apps\Datical\repl
cd XYZportal_ddb
call hammer forecast %DBDEF%
call hammer statusDetails %DBDEF%
call hammer deploy %DBDEF%
call hammer statusDetails %DBDEF%

Deploy Report

The deploy reports are located in the project directory, typically: <project_dir>/Reports/<YYYY>/<DD-Mon>/<dbDef> folders. There are subdirectories for each deploy report that is invoked which contain the HTML report.
