dynamodb-waiters-enabled

--dynamodb-waiters-enabled is a global Boolean parameter for the DynamoDB Pro extension. It specifies whether to enable or disable the DynamoDbWaiter for all DynamoDB operations. The default value is true.

When true, waits for DynamoDB operations in a to be fully complete before moving onto the next . When false, moves onto the next while the first one may still be in a pending state, potentially saving time. When false, also suppresses all other --dynamodb-waiter-* parameters.

If you set this parameter to false, you risk unexpected behavior. You must ensure that any s you deploy do not try to access objects that are in a pending state.

The  Tracking Tables ( and , and ) always use waiters, regardless of the value you specify for --dynamodb-waiters-enabled.

Uses

The --dynamodb-waiters-enabled parameter controls whether the DynamoDB waiter is enabled for operations:

  • If you want to wait for each to finish deploying in DynamoDB before running the next , no matter how long the DynamoDB operations take, leave the parameter at its default value of true. This ensures that your s cannot unintentionally access objects in a pending state, but may take longer.
  • If you want to not wait for each to finish deploying in DynamoDB before running the next , set the parameter to false. This ensures that your CI/CD workflow moves forward rapidly, but may result in unexpected behavior.

    • If you set this parameter to false, you must be certain that the later changes in your will not try to access objects modified earlier in your (which may still be running and may be in a pending state).

Troubleshooting

If you disable waiters, may sometimes attempt to modify a pending resource:

Unexpected error running Liquibase: Migration failed for changeset changelogs/example-changelog.xml::2::your.name:
     Reason: liquibase.exception.DatabaseException: Could not execute
  - Caused by: Attempt to change a resource which is still in use: Table is being created: Music (Service: DynamoDb, Status Code: 400, Request ID: 61FCQO19S8P91M5TD8RCTAU60NVV4KQNSO5AEMVJF66Q9ASUAAJG)

If you experience this issue, you should re-run your failed change after waiting enough time for the resource to be in an ACTIVE state. Alternatively, you can re-enable waiters and set higher wait times if necessary.
