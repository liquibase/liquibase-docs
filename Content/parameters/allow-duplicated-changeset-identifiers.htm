allow-duplicated-changeset-identifiers

--allow-duplicated-changeset-identifiers is a global Boolean parameter. You can use it to specify whether lets you deploy multiple s with the same filepath, author, and ID. It is available in 4.25.1 and later. The default value is false.

Uses

By default, requires every to have a unique identifier, which is the filepath of the combined with the author and ID of the . This ensures that each of your changes are clear and distinct.

When you deploy a , creates a row for that in your [%=General.databasechangelog%] table. If any s contain duplicate identifiers (which would create duplicate rows in the DBCL table), halts the deployment:

Unexpected error running Liquibase: Validation Failed:
     1 changesets had duplicate identifiers
          example-changelog.xml::duplication-test::liquibase-user

However, your may contain s with duplicated identifiers. Developers may have manually added these s out of the normal process. If your is very large, it may be unfeasible to remove them manually. To ignore the duplicates, set --allow-duplicated-changeset-identifiers to true.

This way, only creates a row in your table for the first with a duplicated identifier. For every following with that duplicated identifier, does not create a new row. This means your duplicated s are not deployed to your database, but will not treat the duplicates like errors, so it will not halt during the deployment.

Reusing identifiers can be confusing to developers and may lead to unexpected rollback behavior. It is a best practice to use distinct identifiers for every in .

Rolling back changes with the same identifier

If you have two s with the same identifier, you can use the rollback, rollback-count, rollback-one-update, and rollback-to-date commands normally. For these commands, there is no difference in rolling back a with a duplicated identifier or a unique identifier.

However, the rollback-one-changeset command only targets the first because it is the only one that appears in the table. When you run rollback-one-changeset and specify your duplicated identifier, rolls back only the first . To avoid checksum errors, you must also specify --allow-duplicated-changeset-identifiers=true when running this rollback command. For example:

liquibase
    --allow-duplicated-changeset-identifiers=true
  rollback-one-changeset
    --changeset-path=example-changelog.xml
    --changeset-author=liquibase-user
    --changeset-id=duplication-test
    --force
