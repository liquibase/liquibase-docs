Using with Databricks Platforms

Databricks SQL is a fully managed analytics data warehouse that enables users to run SQL queries on large datasets stored in cloud-based data lakes. It is designed for data analysts and engineers who need to perform data exploration, transformation, and analysis. For more information, see Databricks SQL Documentation.

You can deploy most standard  [%=General.changetypes%]s to Databricks. supports Databricks-specific functionality with some additional s available in the extension for Databricks. For a list of supported features, see the "Features" section.

offers two extensions for Databricks: a commercial (Pro) extension for users and an open-source (OSS) extension for users. The Databricks Pro extension contains all functionality in the Databricks OSS extension, as well as many additional features. To access features, you need a license key.

Features

Supported features include:

  • Policy Checks: automatically analyze your s for desired format and behavior to increase deployment success rates and uphold security best practices
  • Secrets Management: keep your authentication data secure by integrating with third-party secrets vaults
  • [%=General.StructuredLogging%]: improve your database observability by easily reading data in your favorite analytics tool
  • Operation Reports: generate reports of operations you perform on your database
  • Flow Files: create repeatable, portable, and platform-independent workflows to run in any of your CI/CD tools
  • [%=General.databasechangeloghistory%] table (DBCLH): record a history of all changes you make to the database, including changes that are not tracked by the table (DBCL)
  • Remote file access: centralize file management with AWS S3 to build a reusable repository of files you can update and retrieve
  • Targeted rollback: avoid collateral damage by specifying which s in your to undo
  • Stored Logic: capture stored logic objects like functions

Supported parameters

You can use all normal [%=General.Liquibase%] Parameters with Databricks. The extension for Databricks offers additional parameters for more advanced functionality.

The following parameters, including databricks-catalog and databricks-schema, only work with OIDC authentication. They will not work for token authentication.

To configure Databricks-specific behavior, set the following parameters in the CLI, in flow files, in your file, or as environment variables:

  • --databricks-auth-mechanism
  • --databricks-catalog
  • --databricks-http-path
  • --databricks-oauth-auth-flow
  • --databricks-oauth-client-id
  • --databricks-oauth-client-secret
  • --databricks-schema

For more information, see [%=General.Liquibase%] Parameters for Databricks.

Limitations

  •  does not support the followings:
    • addAutoIncrement
    • addUniqueConstraint
    • createIndex
    • createPackage
    • createPackageBody
    • createProcedure
    • createSequence
    • createTrigger
    • dropPackage
    • dropPackageBody
    • dropUniqueConstraint
    • modifyDataType
    • renameTable
  • You can deploy changes to Databricks using formatted SQL, YAML, JSON, and XML s, but not unformatted SQL s. To write SQL directly, you can use the sql and sqlFile s.
  •  does not include the values of tableFormat and partitionColumns in files generated by the database inspection commands diff-changelog and generate-changelog. Databricks cannot add new partition columns to an existing table, nor can it change the table's format or location. Running these s on existing tables would cause failures, so omits them from s it generates.
    • However, the outputs of diff and snapshot do include tableFormat and partitionColumns.
  •  does not include all auto-increment (identity) information in files generated by diff-changelog. This information is omitted when a table has the same name in both the source and target databases, but different values for autoIncrement. Also, the addAutoIncrement  is not supported for Databricks.  
    • However, files generated by the diff command do include auto-increment information even when the tables have different values for autoIncrement.
  • Databricks does not support directly adding a new column with a default value to a table that already contains columns with default values. In Databricks 1.4.0, you cannot use the addDefaultValue  in an YAML, JSON, or XML  to add a default value to an existing Databricks table column. Instead, you must specify the SQL queries for addDefaultValue in a Formatted SQL  or use the sql or sqlFile s in a YAML, JSON, or XML . For syntax examples, see the "Troubleshooting" section of the addDefaultValue page.
